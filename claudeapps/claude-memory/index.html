CLAUDE MEMORY

SESSIONÜBERGREIFENDES GEDÄCHTNIS · RECHNER · GITHUB

Was ist das Problem?

Claude hat kein eingebautes Langzeitgedächtnis. Jede neue Session startet kalt – Claude weiß zunächst nicht, was in früheren Gesprächen besprochen wurde, welche Architekturentscheidungen getroffen wurden oder wie du am liebsten zusammenarbeitest.

Dieses Problem wird durch ein Zusammenspiel aus lokalen Memory-Dateien, dem Repo (CLAUDE.md + Git-History) und GitHub gelöst.

Dieses Dokument beschreibt, wie das konkret funktioniert – damit du verstehst, was Claude zwischen Sessions „weiß" und was nicht, und warum das auf einem anderen Rechner eventuell anders ist.

Zusammenspiel im Überblick

wird beim Session-Start gelesen wird in Memory geschrieben Git-Sync zwischen Rechnern ✗ = nicht verfügbar

Die vier Memory-Typen

Claude speichert Informationen kategorisiert in ~/.claude/projects/…/memory/. Der Index MEMORY.md wird bei jeder Session automatisch geladen.

Typ Inhalt Beispiel
user Rolle, Präferenzen, Wissensstand des Nutzers „Nutzer bevorzugt Ein-Kommando-Lösungen, wenig CLI-Erfahrung"
feedback Korrekturen und bestätigte Entscheidungen „Keine Erklärung von Basiskonzepten, direkt zur Lösung"
project Laufende Ziele, Deadlines, Kontext „Upgrade-Stufe 4 als nächstes, Stufen 1–3 abgeschlossen"
reference Zeiger auf externe Systeme & Ressourcen „Odoo-Analyzer unter backend.kasimir.info/analyzer"

Was Claude in einer neuen Session weiß

🗂️

Memory-Dateien

Gespeicherte Nutzer-Präferenzen, Feedback und Projektstand – sofern auf diesem Rechner vorhanden.

📄

CLAUDE.md

Liegt im Repo, ist auf jedem Rechner nach git pull verfügbar. Enthält Projektregeln, Struktur und Konventionen.

📜

Git-History

Alle Commits mit Autor, Datum und Beschreibung. Claude kann nachvollziehen, was wann gebaut wurde – über Rechner hinweg.

💻

Codebase

Alle Quelldateien inkl. Plan-Dokumente im Repo (z.B. UPGRADE_ANALYZER_PLAN.md). Offene TODOs sind direkt lesbar.

Was Claude nicht weiß: Den genauen Gesprächsverlauf früherer Sessions, nicht-committete Änderungen auf einem anderen Rechner, und die lokale Memory-Datei eines anderen Rechners.

Rechner-Wechsel: Was passiert?

Szenario: Du arbeitest auf Rechner 1, wechselst auf Rechner 2

  • Code, CLAUDE.md, git log → sofort verfügbar nach git pull ✓
  • Memory-Dateien (~/.claude/…) → fehlen auf Rechner 2
  • Claude baut den Kontext neu auf: liest CLAUDE.md, git log, Plan-Docs im Repo
  • Präferenzen aus Memory müssen ggf. kurz wiederholt werden (einmalig pro Rechner)
Tipp: Wichtige Projektregeln und Konventionen gehören in CLAUDE.md im Repo – dann sind sie auf jedem Rechner sofort nach git pull aktiv, ohne Memory-Dateien.

Stabiles Gedächtnis ohne Rechner-Abhängigkeit

Alles, was in Git gepusht ist, steht Claude überall zur Verfügung:

CLAUDE.md # Projektregeln → in jedem Repo-Root docs/PLAN.md # Pläne + offene TODOs → lesbar per git pull Commit-Messages # Was wann entschieden wurde → git log

Wo liegen die Memory-Dateien?

Die Memory-Dateien sind projektspezifisch und liegen lokal auf dem jeweiligen Rechner:

~/.claude/projects/<pfad-zum-projekt>/memory/ ├── MEMORY.md ← Index, wird bei jeder Session geladen ├── user_role.md ├── feedback_style.md ├── project_upgrade.md └── ...
Diese Dateien sind nicht Teil des Git-Repos – sie enthalten persönliche Präferenzen und Projektnotizen, die nicht ins Repo gehören. Daher werden sie nicht mit GitHub synchronisiert.

Claude schreibt automatisch in diese Dateien, wenn in einer Session etwas Wichtiges auftritt – z.B. eine Korrektur, eine neue Präferenz oder ein Projektstand-Update.
