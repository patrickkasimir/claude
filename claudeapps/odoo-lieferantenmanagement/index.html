Lieferanten­management

Odoo 19.3 · SOLVVision

Wofür ist das Lieferantenmanagement?

Die App bündelt alles rund um die Bewertung und Verwaltung von Lieferanten an einem Ort: Verträge, eine Wesentlichkeitsbeurteilung nach ISO 27001 (wie kritisch ist ein Lieferant für die Informationssicherheit?), eine Qualitätsbewertung (wie gut liefert er?) und ein Risikoregister. Zuvor lief das in einer Excel-Tabelle – jetzt strukturiert in Odoo, mit automatischen Erinnerungen und nachvollziehbarer Historie.

So läuft der Prozess (Use-Cases)

1. Einen Lieferanten ins Management aufnehmen

  • In Kontakte den Lieferanten suchen oder neu anlegen.
  • Im Tab „Lieferantenmanagement" die Kategorie setzen (Strategisch oder Operativ/Transaktional) und – nach Bedarf – Bereich, Owner, Datenschutz-Klasse, Zertifikate.
  • Sobald die Kategorie gesetzt ist, erscheint der Lieferant automatisch im Menü „Lieferanten". Das ist dieselbe Kontakt-Akte, nur gefiltert dargestellt – nichts wird kopiert.

2. Regelmäßig bewerten lassen

  • Eine Bewertungsaufforderung anlegen: Typ (Wesentlichkeit oder Qualität), Turnus (jährlich / halbjährlich / quartalsweise oder einmalig) und den Geltungsbereich (einzelne Lieferanten zum Testen oder ein Filter).
  • Oben am Datensatz steht die Phase als Statusleiste: Entwurf → Aktiv → Inaktiv. Erst wenn die Phase auf Aktiv gesetzt wird, wird ausgelöst – so kann man eine Aufforderung in Ruhe als Entwurf vorbereiten.
  • Bei wiederkehrendem Turnus erzeugt das System zur Fälligkeit für jeden betroffenen Lieferanten eine Bewertung und legt dem Owner eine Aufgabe an; danach rückt die nächste Fälligkeit um den Turnus weiter. Bei einmalig geschieht das genau einmal direkt beim Aktivieren. Zum sofortigen Anstoßen gibt es zusätzlich den Button „Jetzt auslösen".
  • Der Owner öffnet die Bewertung, wählt je Kriterium einen Wert (0–3) – Score und Einstufung berechnen sich automatisch.

3. Risiken behandeln

  • Fällt eine Bewertung hoch aus, kann sie per Button „In Risikoregister überführen" übernommen werden.
  • Im Risikoregister werden Ursache, Maßnahmen und Überwachung gepflegt – entkoppelt von der einzelnen Bewertung.

Wie wird gerechnet?

Jedes Kriterium bekommt 0–3 Punkte. Jeder Punkt wird mit einem Gewicht multipliziert; die Summe ist der Gesamtscore (0–3), der automatisch zu einer Einstufung führt.

Wesentlichkeitsbeurteilung (ISO 27001)

KriteriumGewicht
Zugriff auf vertrauliche Daten25 %
Kritikalität für Geschäftsprozesse20 %
IT-Sicherheit & Compliance20 %
Standort & rechtliche Rahmenbedingungen15 %
Historie & Zuverlässigkeit10 %
Vertragsgestaltung10 %

Einstufung: < 1,0 nicht wesentlich 1,0 – 1,9 mittel ≥ 2,0 wesentlich / hoch

Beispiel: Punkte 3 / 3 / 2 / 1 / 1 / 1 → 3·0,25 + 3·0,20 + 2·0,20 + 1·0,15 + 1·0,10 + 1·0,10 = 2,10 → wesentlich / hoch

Qualitätsbewertung

KriteriumGewicht
Qualität25 %
Zuverlässigkeit25 %
Lieferfähigkeit25 %
Service / Kundenorientierung15 %
Kosten10 %

Einstufung: ≤ 2,0 gering 2,1 – 2,3 mittel > 2,3 hoch

Hinweis: Bei den Qualitätskriterien bedeutet ein höherer Punktwert ein größeres Risiko (3 = „unter der Anforderung").

Überblick & Vorgehen

Bei Odoo Online gibt es keinen Server-Zugriff und man kann kein eigenes Python-Modul installieren. Die gesamte App wurde daher rein über die Standard-Schnittstelle XML-RPC von außen konfiguriert – Studio-kompatibel (Felder mit x_-Präfix), wiederholbar über nummerierte, idempotente Python-Skripte, zweisprachig DE/EN.

Architektur & Datenfluss

Linux-Server (Backend) Odoo-Instanz (SOLVVision)

Datenmodell

Lieferant (res.partner) Steuerung & Bewertungen Risiko Cron / Vertrag

Die „Lieferanten"-Sicht ist eine reine, gefilterte Ansicht auf res.partner (Domain is_company = True und Kategorie gesetzt) – keine Datenkopie. Klassifizierungsfelder (x_studio_lm_*) liegen direkt am Kontakt.

Berechnung, Cron & Server-Actions

  • Score/Einstufung als manuelles Compute-Field (Code in ir.model.fields.compute, Item-Zuweisung record['x_feld'] = …); Gewichte/Schwellen wie in der Nutzerdoku.
  • Phase der Aufforderung (Entwurf / Aktiv / Inaktiv, klickbare Statusbar): ausgelöst wird nur bei Phase = Aktiv.
  • Cron „LM: Bewertungen auslösen" (täglich): findet wiederkehrende Aufforderungen mit Phase = Aktiv und nächste Fälligkeit ≤ heute, legt je Lieferant eine Bewertung an, erzeugt die To-Do-Activity und schiebt die Fälligkeit um den Turnus weiter (gerechnet ab der bisherigen Fälligkeit).
  • Einmalig: eine base.automation löst beim Setzen der Phase auf Aktiv genau einmal aus (Guard über letzter Lauf); der tägliche Cron lässt einmalige Aufforderungen aus.
  • „Jetzt auslösen" (Server-Action am Aufforderungs-Formular): dieselbe Logik sofort, ohne Fälligkeitsprüfung und ohne den Turnus zu verschieben.
  • Zielmenge: ist das M2M-Feld x_studio_filter_partner_ids gesetzt, werden genau diese Lieferanten bewertet (übersteuert Kategorie/Bereich/„nur wesentliche").
  • „In Risikoregister überführen": Server-Action legt einen Risikoeintrag an und verlinkt ihn; Einstufung wird gemappt.

Besonderheiten in Odoo 19.3 (verifiziert)

  • Eigene Felder müssen mit x_ beginnen; Auswahlwerte als Selection mit Sequenz.
  • Übersetzbare Felder immer mit explizitem Sprach-Kontext (context={'lang': …}).
  • ir.cron kennt das Feld numbercall in 19.3 nicht mehr – weglassen.
  • Im Code von Server-Actions ist fields nicht verfügbar (anders als bei Compute-Feldern) → datetime.date.today() statt fields.Date.today().
  • Menü-/Action-Namen sind übersetzbar: für idempotente Skripte müssen Suche und Schreiben im de_DE-Kontext erfolgen, sonst entstehen Duplikate.
  • Chatter/Aktivitäten via is_mail_thread / is_mail_activity; Listen heißen <list>, Chatter <chatter/>.
