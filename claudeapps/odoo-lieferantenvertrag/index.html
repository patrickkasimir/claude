Lieferantenvertrags­management

Odoo 19 · Customizing über die External API (XML-RPC) · SOLVVision

Worum geht es?

Für die SOLVVision-Odoo-Instanz (Odoo 19, Online/SaaS) wurde ein Studio-kompatibles Datenmodell für Lieferantenverträge aufgebaut – Vertragsarten AVV, NDA, Rahmenvertrag und Einzelvereinbarung, mehrsprachig, mit Verknüpfung zum Einkauf, Automationen und PDF-Vorlagen.

Das Besondere: Bei Odoo Online gibt es keinen Server-Zugriff und man kann kein eigenes Python-Modul installieren. Alles wurde daher rein über die Standard-Schnittstelle XML-RPC von außen konfiguriert – wiederholbar über nummerierte, idempotente Skripte.

Architektur & Datenfluss

Wie das Zusammenspiel von Steuerung, Linux-Server und Odoo-Instanz funktioniert:

Linux-Server (mein Backend) Odoo-Instanz (SOLVVision)

So läuft es Schritt für Schritt

  • Auftrag: Du gibst im Chat eine Anweisung (vom Mac aus).
  • Claude Code läuft auf dem Linux-Server (nicht auf dem Mac) und steuert alles von dort.
  • Über das Bash-Tool führt es die Python-Skripte aus; gemeinsamer Kern ist odoo_client.py.
  • Die Skripte lesen die Zugangsdaten aus .env (nie im Git, verlässt den Server nicht); pdf_gen.py erzeugt die PDF-Vorlagen lokal.
  • Per XML-RPC über HTTPS (authenticate + execute_kw) wird die Odoo-Instanz angesprochen – schreiben und zurücklesen zur Prüfung.
  • In Odoo entstehen Datenmodell, Felder, Views, Menü, Automationen und die Daten.

Zusammenspiel: Vertrag ↔ Bestellung

Eine Einzelvereinbarung legt keine Bestellung an – die Beziehung läuft andersherum: Die Bestellung verweist auf den Vertrag, eine Automation trägt sie dann im Vertrag ein.

Eine zweite Automation erinnert 30 Tage vor Vertragsende per Aktivität an den Verantwortlichen – auch sie erzeugt keine Bestellung.

Besonderheiten in Odoo 19 (verifiziert)

  • translate ist jetzt ein Auswahlwert (standard/html_translate/xml_translate), kein Ja/Nein mehr.
  • Übersetzbare Felder immer mit explizitem Sprach-Kontext (context={'lang': …}) lesen/schreiben.
  • Pflicht-Beziehungsfelder brauchen on_delete='restrict' (nicht „auf null setzen“).
  • Code in Automationen darf keine Attribute direkt setzen – rec.write({…}) statt rec.feld = ….
  • Listen heißen <list> (nicht mehr <tree>); Chatter via <chatter/>.
  • Eigene Felder müssen mit x_ beginnen – deshalb ist die Rechnungs-PDF-Vorschau (rechts) auf einem Studio-Modell nicht möglich; Ersatz ist das pdf_viewer-Widget.
  • Eine Bestellung im Status „Entwurf“ ist eine Angebotsanfrage; das Menü „Bestellungen“ zeigt nur bestätigte – daher werden Demo-Bestellungen bestätigt.
